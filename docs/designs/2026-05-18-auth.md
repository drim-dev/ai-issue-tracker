# Дизайн: модуль Auth

**Дата:** 2026-05-18
**Статус:** утверждён

## Цель

Аутентификация по email/паролю для AI Issue Tracker: регистрация, логин, логаут
и эндпоинт «кто я». Auth — первая фича проекта, поэтому реализуется вместе со
скелетом решения (Aspire AppHost, ServiceDefaults, Api с EF Core + Postgres,
Next.js-приложение). Скелет детализируется в плане реализации; этот документ —
дизайн самого Auth.

## Решения

| Вопрос | Решение |
|--------|---------|
| Хеширование паролей | Argon2id через `Isopoh.Cryptography.Argon2` |
| Модель сессии | Шифрованная stateless-cookie (`iron-session`), без хранилища |
| Payload cookie | Только `{ userId }` |
| TTL сессии | 7 дней, sliding, значение в конфиге |
| OAuth / «remember me» | Не делаем (вне MVP / YAGNI) |

## Контур и операции

```
[Браузер]  <--cookie-->  [BFF / Next.js]  <--X-User-Id header-->  [.NET API]
```

- **Граница браузер ↔ BFF** — сессия в httpOnly + Secure + SameSite=Lax cookie.
- **Граница BFF ↔ .NET API** — доверенный заголовок `X-User-Id`; токена нет,
  API не опубликован наружу и принимает трафик только от BFF внутри сети Aspire.

| Операция | .NET API | BFF route handler | Аноним |
|----------|----------|-------------------|--------|
| Регистрация | `POST /auth/register` | `POST /api/auth/register` | да |
| Логин | `POST /auth/login` | `POST /api/auth/login` | да |
| Логаут | — | `POST /api/auth/logout` | нет |
| «Кто я» | `GET /auth/me` | — (читает Server Component напрямую) | нет |

- **Логаут — чисто на BFF**: уничтожает сессионную cookie. API stateless,
  отзывать нечего.
- **`/auth/me`** на .NET API возвращает свежий `User`-DTO по `X-User-Id`.
  Отдельный BFF-роут не нужен — Server Components ходят в API через серверный
  fetch-хелпер.
- **register/login — анонимные** эндпоинты API.

## Домен и бэкенд

### Сущность `User` (`Domain/User.cs`)

| Поле | Тип | Примечание |
|------|-----|------------|
| `Id` | `long` | IdGen (скилл id-generation), не `Guid` |
| `Email` | `string` | уникальный индекс, хранится в lower-case |
| `PasswordHash` | `string` | encoded-строка Argon2 (соль + параметры внутри) |
| `Name` | `string` | |
| `Avatar` | `string?` | URL, в Auth-фиче не задаётся |
| `CreatedAt` | `DateTime` | UTC |

### Хеширование

- Абстракция `IPasswordHasher` в `Common/Auth/`: `Hash(password)`,
  `Verify(password, hash)`.
- Реализация `Argon2PasswordHasher` через `Isopoh.Cryptography.Argon2`
  (Argon2id, self-contained encoded-строка). Параметры — константы в
  реализации; Options-паттерн избыточен.

### `UserContextMiddleware` (`Common/Auth/`)

Читает заголовок `X-User-Id` → кладёт `ClaimsPrincipal` с `NameIdentifier`.
Нет заголовка — principal анонимный. `/auth/me` требует аутентификации;
register/login — `AllowAnonymous`.

### Вертикальные слайсы (`Features/Auth/`)

- `Register` — нормализует email в lower-case, проверяет уникальность,
  хеширует пароль, создаёт `User` с IdGen-id, сохраняет, возвращает DTO.
- `Login` — ищет user по email, `Verify` пароля; успех → DTO;
  провал → доменное исключение → **401**.
- `GetCurrentUser` — берёт id из claims, грузит `User`, возвращает DTO;
  нет пользователя → 404.

### DTO

`UserResponse { id, email, name, avatar }` — наружу никогда не отдаём
EF-сущность и никогда `PasswordHash`.

## BFF, сессия, потоки данных

### Сессия (`lib/auth/session.ts`)

Конфиг `iron-session`: cookie `ai_tracker_session`, флаги
`httpOnly + Secure + SameSite=Lax`, секрет из env `SESSION_SECRET`,
`ttl` 7 дней из конфига, sliding (cookie продлевается через хелпер).
Payload: `{ userId: string }`.

### Route handlers BFF

- `POST /api/auth/register` — Zod-валидация → fetch `.NET /auth/register` →
  успех: `session.userId = id; session.save()` → `200`; `ProblemDetails`
  пробрасывается как есть.
- `POST /api/auth/login` — fetch `.NET /auth/login` → успех: ставим сессию →
  `200`; `401` пробрасывается.
- `POST /api/auth/logout` — `session.destroy()` → `200`.

### Серверный API-хелпер (`lib/api/`)

Читает сессию, подставляет `X-User-Id` в запросы к .NET API. Без сессии
заголовок не ставится (для `/auth/me` это даст 401).

### `middleware.ts`

- Нет сессии и маршрут защищён → редирект на `/login`.
- Есть сессия и пользователь на `(auth)`-страницах → редирект на `/projects`.

### Потоки

```
Регистрация: форма → POST /api/auth/register → Zod → .NET /auth/register
             → 201+User → iron-session.save({userId}) → редирект /projects

Логин:       форма → POST /api/auth/login → .NET /auth/login
             → 200+User → iron-session.save({userId}) → редирект /projects

Логаут:      кнопка → POST /api/auth/logout → session.destroy() → редирект /login

Защищённый:  Server Component → apiFetch('/auth/me')
             → хелпер добавляет X-User-Id → .NET API → User-DTO
```

## UI (`app/(auth)/`)

- `login/page.tsx`, `register/page.tsx` — формы на React Hook Form +
  `zodResolver`, компоненты shadcn/ui, только токены `DESIGN-SYSTEM.md`.
- Поля register: `email`, `name`, `password`, `confirmPassword`
  (`confirmPassword` проверяется только на клиенте). Login: `email`, `password`.
- Логаут — кнопка в шапке приложения (`POST /api/auth/logout`).
- Zod-схемы в `lib/validations/` зеркалируют правила FluentValidation.

## Валидация

| Поле | Правило |
|------|---------|
| `email` | формат email |
| `name` | не пусто, разумный лимит длины |
| `password` | мин. длина 8 |

Backend (FluentValidation) и frontend (Zod) держим в синхроне.

## Обработка ошибок (`ProblemDetails`)

| Ситуация | HTTP | Поведение формы |
|----------|------|-----------------|
| Невалидный ввод | 400 + `errors` | раскладка по полям |
| Email уже занят | 409 + `errorCode` | ошибка на поле `email` |
| Неверный email/пароль | 401 | общая ошибка формы «Неверный email или пароль» (не раскрываем, существует ли email) |
| Нет сессии на защищённом маршруте | — | редирект на `/login` (middleware) |

## Тестирование

- **Компонентные** (TestContainers + harness, скилл `component-testing`):
  register — успех, дубль email, невалидный ввод; login — успех, неверный
  пароль, неизвестный email; `/auth/me` — с заголовком и без.
- **Валидация** — отдельно через `FluentValidation.TestHelper`.
- **Фронтенд** — Jest + RTL: формы login/register, раскладка ошибок по полям.

## Вне зоны (YAGNI)

OAuth, сброс/восстановление пароля, смена пароля, редактирование профиля,
подтверждение email, «remember me», серверное хранилище сессий, отзыв сессий.
