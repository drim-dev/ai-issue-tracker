# Auth

Аутентификация по email/паролю для AI Issue Tracker: регистрация, логин, логаут и
эндпоинт «кто я». Браузер общается только с BFF (Next.js), который держит сессию в
шифрованной cookie; .NET API доверяет заголовку `X-User-Id` от BFF.

## Модель данных

| Сущность | Ключевые поля | Примечания |
|----------|--------------|------------|
| `User` | `Id: long`, `Email`, `PasswordHash`, `Name`, `Avatar?`, `CreatedAt` | `Email` — уникальный индекс, хранится в lower-case, служит логином. `PasswordHash` — encoded-строка Argon2id (соль и параметры внутри). `Avatar` — URL, в Auth не задаётся. `CreatedAt` — UTC. |

## API-эндпоинты

### .NET API

| Метод | Путь | Авторизация | Описание |
|-------|------|-------------|----------|
| POST | `/auth/register` | Anonymous | Регистрирует пользователя, возвращает `UserResponse` со статусом `201`. |
| POST | `/auth/login` | Anonymous | Проверяет пароль, возвращает `UserResponse` со статусом `200`. |
| GET | `/auth/me` | Authenticated | Возвращает свежий `UserResponse` по `X-User-Id`. |

### BFF route handlers

| Метод | Путь | Авторизация | Описание |
|-------|------|-------------|----------|
| POST | `/api/auth/register` | Anonymous | Проксирует в `/auth/register`, при успехе ставит сессию. |
| POST | `/api/auth/login` | Anonymous | Проксирует в `/auth/login`, при успехе ставит сессию. |
| POST | `/api/auth/logout` | Authenticated | Уничтожает сессионную cookie. |

`UserResponse { id, email, name, avatar }` — наружу никогда не отдаётся EF-сущность
или `PasswordHash`.

## Ключевые поведения

- **Хеширование** — Argon2id; абстракция `IPasswordHasher` (`Hash`, `Verify`),
  параметры зашиты в реализацию.
- **Нормализация email** — register и login приводят email к lower-case перед
  поиском и сохранением.
- **Уникальность email** — гарантируется уникальным индексом БД; при гонке
  нарушение constraint мапится в `409`.
- **Логин** — провал проверки email или пароля даёт `401` с общим сообщением,
  не раскрывающим, существует ли email.
- **`/auth/me`** — id берётся из claims; если пользователь удалён при живой
  сессии → `404`.
- **Контекст пользователя** — `X-User-Id` от BFF превращается в `ClaimsPrincipal`
  с `NameIdentifier`; отсутствие заголовка → анонимный principal.
- **Сессия** — шифрованная stateless-cookie `ai_tracker_session`
  (`httpOnly + Secure + SameSite=Lax`), payload `{ userId }`, TTL 7 дней.
- **Sliding-продление** — серверный API-хелпер при каждом запросе с валидной
  сессией пересохраняет cookie, сдвигая TTL на 7 дней вперёд.
- **Логаут** — выполняется только на BFF (уничтожение cookie); API stateless,
  отзывать нечего.
- **Защита маршрутов** — middleware редиректит неаутентифицированных с защищённых
  маршрутов на `/login`, а аутентифицированных с `(auth)`-страниц на `/projects`.
- **Протухшая сессия** — `404` на `/auth/me` Server Component трактует как
  невалидную сессию и редиректит на `/login`.

## Валидация

| Поле | Правило |
|------|---------|
| `email` | формат email |
| `name` | не пусто, лимит длины |
| `password` | мин. длина 8 |
| `confirmPassword` | совпадает с `password` — только на клиенте |

Backend (FluentValidation) и frontend (Zod) держатся в синхроне.

## Обработка ошибок

| Ситуация | HTTP | Поведение формы |
|----------|------|-----------------|
| Невалидный ввод | `400` + `errors` | раскладка по полям |
| Email уже занят | `409` + `errorCodes` | ошибка на поле `email` |
| Неверный email/пароль | `401` | общая ошибка формы «Неверный email или пароль» |
| Нет сессии на защищённом маршруте | — | редирект на `/login` (middleware) |
| Сессия валидна, пользователь удалён | `404` на `/auth/me` | редирект на `/login` |

## Вне зоны

OAuth, сброс/восстановление и смена пароля, редактирование профиля, подтверждение
email, «remember me», серверное хранилище сессий, отзыв сессий.
