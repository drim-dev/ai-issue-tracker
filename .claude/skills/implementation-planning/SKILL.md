# Планирование реализации

Создай поэтапный план реализации из документа дизайна, структурированный для исполнения командой агентов Claude Code.

## Контекст: как работают команды агентов

Команды агентов Claude Code состоят из **team lead** (основная сессия) и **членов команды** (спавнящиеся экземпляры Claude Code). Ключевые свойства:

- Все члены команды используют **одну рабочую директорию** (без изоляции)
- У членов команды свои окна контекста, но они загружают тот же CLAUDE.md, скиллы, MCP-серверы
- Члены команды общаются через **общий список задач** и **прямые сообщения**
- У задач есть **зависимости** — заблокированные задачи ждут, пока завершатся их зависимости
- Члены команды **сами берут** доступные задачи или получают их от лида
- **Конфликты файлов — главный риск**: два члена команды, редактирующих один и тот же файл, перезаписывают изменения друг друга

## Когда использовать

После того как документ дизайна написан и утверждён. До написания любого кода реализации.

## Процесс

### 1. Анализ дизайна

- Прочитай документ дизайна тщательно
- Определи все дискретные единицы работы
- Построй карту зависимостей между ними (что что блокирует)
- Определи, какие единицы затрагивают backend, frontend или оба

### 2. Построение графа зависимостей

Для каждой единицы работы определи:
- Что ей нужно перед стартом (входы / предусловия)
- Что она производит, что нужно другим (выходы)
- Является ли она только backend, только frontend или сквозной

Единицы без зависимостей между собой идут в одну фазу (параллельно).

### 3. Написание плана

Создай `docs/plans/YYYY-MM-DD-<topic>.md` со структурой ниже.

#### Структура плана

```markdown
# План реализации: <Тема>

Дизайн: `docs/designs/YYYY-MM-DD-<topic>.md`

## Фаза 1 — <Название фазы>

*Без зависимостей. Все члены команды работают параллельно.*

### Задача 1.1: <Краткое название задачи>
- [ ] Описание шага 1
- [ ] Описание шага 2
- [ ] Описание шага 3

**Файлы:** `path/to/file1.ts`, `path/to/file2.cs`

### Задача 1.2: <Краткое название задачи>
- [ ] Описание шага 1
- [ ] Описание шага 2

**Файлы:** `path/to/file1.ts`, `path/to/file2.ts`

## Фаза 2 — <Название фазы>

*Зависит от: Фазы 1 (причина)*

### Задача 2.1: <Краткое название задачи>
- [ ] Описание шага 1
- [ ] Описание шага 2

**Файлы:** `path/to/file.ts`

...
```

### 4. Модель исполнения

План спроектирован для исполнения **team lead-ом**:

1. **Лид создаёт команду агентов** и общий список задач из плана
2. **Лид спавнит членов команды** — по одному на параллельную задачу в текущей фазе (рекомендуется 3–5 членов команды)
3. **Члены команды сами берут задачи** из общего списка и идут по чекбоксам
4. **У задач есть зависимости** — задачи фазы 2 зависят от задач фазы 1, поэтому остаются заблокированными до завершения фазы 1
5. **Лид мониторит прогресс**, перенаправляет членов команды при необходимости и синтезирует результаты
6. **Лид обновляет чекбоксы** в файле плана по мере завершения задач
7. **Когда все задачи в фазе завершены**, зависимые задачи в следующей фазе автоматически разблокируются
8. **Лид очищает команду**, когда все фазы завершены

Лид должен говорить членам команды следовать релевантным проектным скиллам (например, `vertical-slice-architecture`, `component-testing`) при назначении backend-работы.

## Правила написания плана

### Максимизируй параллелизм
- Чем меньше фаз, тем лучше — переноси единицы работы в более ранние фазы, когда зависимости это позволяют
- Backend и frontend работают с разными файлами, поэтому почти всегда параллелизуются
- Два члена команды могут параллельно работать с разными страницами frontend (разные файлы)
- Два члена команды НЕ МОГУТ безопасно редактировать один и тот же файл — объедини эти единицы в одну задачу или раздели их по разным фазам

### Правила скоупа задач
- Каждая задача — связная, самодостаточная единица работы для одного члена команды
- Задача должна затрагивать сфокусированный набор файлов — перечисли их в разделе **Файлы:**
- Если две задачи будут редактировать один файл, объедини их в одну задачу или разнеси по разным фазам
- Предпочитай небольшие задачи (3–8 чекбоксов) большим — их легче ревьюить и перезапускать при сбое
- Ориентируйся на 5–6 задач на члена команды, чтобы поддерживать их продуктивность

### Гранулярность чекпоинтов
- Каждый чекбокс = одно проверяемое действие (создать файл, добавить функцию, обновить конфиг, запустить тест)
- Слишком гранулярно (каждая строка кода) — это шум; слишком крупно («реализовать фичу») — бесполезно
- Хорошо: «Добавить `generateMetadata` в `/skills/[slug]/page.tsx` с title, description, OG image, canonical URL»
- Плохо: «Обновить страницу скиллов» (слишком расплывчато) или «Добавить строку og:title» (слишком гранулярно)

### Предотвращение конфликтов файлов
- **Перечисляй все файлы, которые затронет каждая задача**, в разделе **Файлы:**
- Перед финализацией плана убедись, что никакие две задачи в одной фазе не разделяют файлы
- Общие утилиты / типы, которые нужны нескольким задачам, должны создаваться в более ранней фазе
- Если общий файл должен редактироваться несколькими задачами, объедини их в одну задачу или разнеси по разным фазам

### Рекомендации по размеру команды
- 3–5 членов команды для большинства работ — баланс между параллелизмом и накладными расходами на координацию
- Каждый член команды может последовательно проходить несколько задач (сам берёт их из списка)
- Больше членов команды ≠ быстрее — стоимость координации растёт, после 5–6 отдача снижается
